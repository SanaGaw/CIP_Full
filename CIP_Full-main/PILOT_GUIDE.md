# CIP v2 Pilot Guide

This guide explains how to run a pilot session of the Collective Intelligence Platform (CIP) version 2 and interpret the outputs. The platform is a work‑in‑progress and this guide will evolve as features are implemented.

## Deployment

### Local Development

To run CIP locally:

1. Install the dependencies using `pip install -r requirements.txt`.
2. Copy `.env.example` to `.env` and provide any necessary API keys.
3. Start the server with:

   ```bash
   uvicorn cip.main:app --reload
   ```

4. Visit `http://localhost:8000` to see the root endpoint. The admin dashboard will eventually live under `/admin` and the facilitator dashboard under `/facilitator`.

### Railway Deployment

The repository includes a `Dockerfile` and `railway.toml` for deploying to [Railway](https://railway.app). To deploy:

1. Sign up for a Railway account and create a new project.
2. Connect your GitHub repository and select this directory for deployment.
3. Set environment variables in the Railway dashboard based on `.env.example`.
4. Deploy the project. Railway will build the Docker image and start the app.

## Running a Session

The current pilot implements only the foundational infrastructure. A full session flow (onboarding → define → divergence → exploration → criteria → evaluation → mapping) requires additional implementation. To experiment with the existing features:

1. Start the server.
2. Connect to the WebSocket endpoint `/ws/{user_id}` using a WebSocket client (e.g. `websocat` or a simple HTML page).
3. Messages sent over the WebSocket are echoed back and logged in the database.
4. Check the `logs/` directory for trace logs and use the exported functions in `observability.py` to generate transcripts.

## Debugging and Observability

Debugging and observability are core to CIP. The `observability.py` module writes events and traces to the database. Use the following helper functions to inspect a session:

- `export_session_transcript(session_id)` – returns a Markdown transcript of all messages.
- `export_session_traces(session_id)` – returns a Markdown dump of all trace records.
- `export_session_replay(session_id)` – returns a JSON object suitable for powering a replay viewer.

## Reading the Report

The final session report will be generated by the Rapporteur agent (not fully implemented). In this pilot the report is a placeholder. Once implemented, the report will summarise the session, analyse idea clusters and tensions, rank options, and recommend next steps.

## Common Issues

- **Database not initialised** – ensure the app has started and run any migrations (`init_db()` is called on startup).
- **API keys missing** – set the relevant environment variables in your `.env` file.
- **Unimplemented features** – many modules contain stubs. Review the build specification to understand what still needs work.

This guide will be updated as the project evolves. Happy piloting!
